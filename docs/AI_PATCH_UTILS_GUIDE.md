# AI Patch Utils Guide

Generalizable utilities for AI-powered JSON Patch (RFC 6902) refinements with automatic fallback.

## Overview

The `ai_patch_utils` module provides reusable tools for using LLMs to generate **targeted changes** to structured data using JSON Patch operations, rather than regenerating entire objects. This approach:

- ✅ **Prevents data loss** - Can't accidentally drop fields
- ✅ **Explicit changes** - Clear audit trail of what changed
- ✅ **Smaller payloads** - Only send/receive deltas
- ✅ **Better validation** - Operations validated before applying
- ✅ **LLM-friendly** - RFC 6902 is well-represented in training data
- ✅ **Automatic fallback** - Falls back to full regeneration if patches fail

## Quick Start

### Basic Usage with PatchRefinementManager

```python
from shared.ai_patch_utils import PatchRefinementManager
from shared.ai_api_client import ai_client

# Your existing config/data
original_data = {
    "model": "claude-sonnet-4-5",
    "batch_size": 10,
    "settings": {
        "timeout": 30,
        "retries": 3
    }
}

# Validation function
def validate_my_data(data):
    errors = []
    if data.get('batch_size', 0) <= 0:
        errors.append("batch_size must be positive")
    return (len(errors) == 0, errors, [])

# Create manager
manager = PatchRefinementManager(
    original_data=original_data,
    validator_fn=validate_my_data,
    ai_client=ai_client,
    model="claude-opus-4-1"
)

# Refine with natural language
result = await manager.refine_with_patches(
    instructions="Use claude-opus-4-6 model and increase batch size to 20",
    fallback_to_full=True,  # Automatically falls back if patches fail
    debug_name="my_refinement"
)

if result.success:
    updated_data = result.updated_data
    print(f"Changes: {result.ai_summary}")
    print(f"Cost: ${result.eliyahu_cost:.6f}")
```

### Manual Patch Application

```python
from shared.ai_patch_utils import apply_patches_with_validation
```

## JSON Patch Format (RFC 6902)

JSON Patch uses an array of operations:

```json
[
  {"op": "replace", "path": "/field", "value": "new_value"},
  {"op": "add", "path": "/new_field", "value": 123},
  {"op": "remove", "path": "/old_field"},
  {"op": "test", "path": "/status", "value": "draft"}
]
```

### Path Syntax (JSON Pointer RFC 6901)

```javascript
"/field"                    // Top-level field
"/nested/field"             // Nested object
"/array/3"                  // Array element (0-indexed)
"/array/3/subfield"         // Nested in array
"/object~1with~0slash"      // Escape special chars (/ and ~)
```

### Operations

| Operation | Purpose | Example |
|-----------|---------|---------|
| `replace` | Change existing field | `{"op": "replace", "path": "/status", "value": "active"}` |
| `add` | Add new field or array element | `{"op": "add", "path": "/tags/2", "value": "urgent"}` |
| `remove` | Delete field or array element | `{"op": "remove", "path": "/temp_field"}` |
| `test` | Verify expected value (safety) | `{"op": "test", "path": "/version", "value": 2}` |
| `move` | Move value to new path | `{"op": "move", "from": "/old", "path": "/new"}` |
| `copy` | Copy value to new path | `{"op": "copy", "from": "/template", "path": "/copy"}` |

## API Reference

### PatchRefinementManager

High-level manager for AI-powered refinements.

```python
manager = PatchRefinementManager(
    original_data: Dict,                    # Data to refine
    validator_fn: Optional[Callable],       # (data) -> (is_valid, errors, warnings)
    schema: Optional[Dict],                 # JSON schema for the data type
    ai_client: Optional[Any],               # AI client instance
    tool_name: str = "refine_with_patches", # Tool name for AI
    model: str = "claude-opus-4-1",         # Model to use
    max_tokens: int = 16000                 # Max tokens
)

result = await manager.refine_with_patches(
    instructions: str,                      # User's refinement request
    context: Optional[Dict[str, str]],      # Additional context sections
    examples: Optional[List[Dict]],         # Example patch operations
    constraints: Optional[List[str]],       # Additional constraints
    fallback_to_full: bool = True,          # Fallback on failure
    fallback_fn: Optional[Callable],        # Custom fallback function
    debug_name: Optional[str]               # Name for logging
) -> RefinementResult
```

**Returns:** `RefinementResult` with fields:
- `success: bool`
- `updated_data: Dict` - Patched data
- `method: str` - "json_patch", "full_replacement", or "failed"
- `patch_operations: List[Dict]` - The patches applied
- `reasoning: str` - AI's explanation
- `ai_summary: str` - Brief description
- `clarifying_questions: str` - Questions for user
- `eliyahu_cost: float` - Cost in USD
- `token_usage: Dict` - Token usage details
- `error: str` - Error message if failed

### apply_patches_with_validation

Low-level function to apply patches safely.

```python
result = apply_patches_with_validation(
    original_data: Dict,
    patch_operations: List[Dict],
    validator_fn: Optional[Callable],
    dry_run: bool = False
) -> PatchResult
```

**Returns:** `PatchResult` with fields:
- `success: bool`
- `patched_data: Dict` - Result if successful
- `error: str` - Error message if failed
- `validation_errors: List[str]` - Validation errors
- `method: str` - "patch", "dry_run", or "failed"

### create_patch_schema

Create a schema for AI responses with patch operations.

```python
schema = create_patch_schema(

```

```python
from shared.ai_patch_utils import PatchRefinementManager
from config_validator import validate_config_complete

async def refine_config(config, instructions, table_analysis):
    def validator_fn(cfg):
        return validate_config_complete(cfg, table_analysis)

    manager = PatchRefinementManager(
        original_data=config,
        validator_fn=validator_fn,
        schema=load_config_schema(),
        ai_client=ai_client
    )

    result = await manager.refine_with_patches(
        instructions=instructions,
        context={
            "Validation Results": "Previous run had 3% fail rate...",

        ],
        constraints=[

    )
```


## Advanced: Fallback Strategies

### Automatic Fallback

```python
async def full_regeneration():
    """Fallback function for full regeneration"""
    # Your full regeneration logic
    return await generate_full_config(...)

result = await manager.refine_with_patches(
    instructions="...",
    fallback_to_full=True,
    fallback_fn=full_regeneration
)

# If patches fail, automatically calls full_regeneration()
```

### Custom Fallback Logic

```python
async def custom_fallback():
    """Custom fallback with retry logic"""
    for attempt in range(3):
        try:
            return await generate_with_simpler_model(...)
        except Exception as e:
            if attempt == 2:
                raise
            await asyncio.sleep(2 ** attempt)

result = await manager.refine_with_patches(
    instructions="...",
    fallback_to_full=True,
    fallback_fn=custom_fallback
)
```

```python
async def generate_config_unified(...):
```

## Best Practices

### 1. When to Use Patches vs Full Regeneration

**Use JSON Patch for:**
- ✅ Refinements of existing configs
- ✅ User requests for specific changes
- ✅ Iterative improvements
- ✅ When preserving structure is important

**Use Full Regeneration for:**
- ✅ New configurations
- ✅ Major restructuring
- ✅ When patches fail validation

### 2. Safety with Test Operations

Always use `test` operations before critical changes:

```python
patches = [
    {"op": "test", "path": "/version", "value": 2},  # Verify version
    {"op": "replace", "path": "/settings/critical_field", "value": "new"},
    {"op": "test", "path": "/settings/critical_field", "value": "new"}  # Verify change
]
```

### 3. Provide Good Context

```python
result = await manager.refine_with_patches(
    instructions="Optimize for speed",
    context={
        "Current Performance": "500ms avg latency",
        "User Feedback": "Too slow on large datasets",
        "Constraints": "Budget: $100/month"
    },
    examples=[...],
    constraints=[
        "latency < 200ms",
        "cost < $100/month"
    ]
)
```

### 4. Validate Thoroughly

```python
def comprehensive_validator(data):
    errors = []
    warnings = []

    # Required fields
    for field in ['model', 'batch_size']:
        if field not in data:
            errors.append(f"Missing required field: {field}")

    # Value constraints
    if data.get('batch_size', 0) <= 0:
        errors.append("batch_size must be positive")

    # Cross-field validation
    if data.get('parallel_calls', 1) > data.get('rate_limit', 100):
        warnings.append("parallel_calls exceeds rate_limit")

    return (len(errors) == 0, errors, warnings)
```

## Limitations

1. **Complex Restructuring** - Patches work best for targeted changes, not major restructuring
2. **Array Reordering** - Reordering arrays requires many operations (consider full regeneration)
3. **LLM Understanding** - LLMs must understand JSON Pointer paths correctly
4. **Path Validation** - Invalid paths fail immediately (use fallback)

## Troubleshooting

### jsonpatch not available

```bash
pip install jsonpatch
```

### Patches fail validation

The manager automatically falls back to full regeneration if `fallback_to_full=True`.

### Invalid path errors

Check that:
- Field names are exact matches
- Array indices are 0-based and exist
- Nested paths are correct (`/parent/child`, not `/parent.child`)

### AI generates wrong patches

Provide better examples and constraints:

```python
examples = [
    {"op": "replace", "path": "/correct/path/example", "value": "value"}
]

constraints = [
    "Path format: /field_name or /nested/field",
    "Array indices are 0-based",
    "Use 'test' before critical changes"
]
```
