# Session ID Management

**Last Updated:** October 22, 2025
**Status:** Production Implementation

---

## Overview

Session IDs uniquely identify user interactions with the system. They are **always generated by the backend**, never by the frontend.

**Key Principle:** Frontend requests operations, backend issues session IDs, frontend stores and reuses them.

---

## Session ID Format

### Standard Format
```
session_YYYYMMDD_HHMMSS_XXXXXXXX
```

**Components:**
- `session_` - Prefix indicating standard session
- `YYYYMMDD` - Date (e.g., 20251022)
- `HHMMSS` - Time (e.g., 143022)
- `XXXXXXXX` - 8-character random hex (from UUID)

**Example:** `session_20251022_143022_a7b3c9d1`

### Specialized Formats

**Table Maker:** Same as standard
```
session_20251022_143022_a7b3c9d1
```

**Config Generation:** Different prefix
```
config_gen_20251022_143530_b2c4d6e8
```

**Demo Testing:** Demo prefix
```
session_demo_20251022_144015_c3e5f7a9
```

## When Session IDs Are Issued

### 1. Table Maker Workflow

**File:** `src/lambdas/interface/actions/table_maker/conversation.py`
**Lines:** 299-304

**Trigger:** User starts a new Table Maker conversation

**Code:**
```python
if not session_id:
    import uuid
    timestamp = datetime.now().strftime('%Y%m%d_%H%M%S')
    random_hex = uuid.uuid4().hex[:8]
    session_id = f"session_{timestamp}_{random_hex}"
    logger.info(f"[TABLE_MAKER] Generated new session ID: {session_id}")
```

**Request:**
```json
{
  "action": "startTableConversation",
  "email": "user@example.com",
  "user_message": "Create a table for AI companies hiring",
  "session_id": null
}
```

**Response:**
```json
{
  "success": true,
  "status": "processing",
  "conversation_id": "table_conv_20251022_143022_c9d1a7",
  "session_id": "session_20251022_143022_a7b3c9d1"
}
```

**Frontend Action:**
```javascript
currentSessionId = data.session_id;  // Store globally
localStorage.setItem('sessionId', data.session_id);  // Persist
```

### 2. Excel Upload Workflow (Validation)

**File:** `src/lambdas/interface/actions/process_excel_unified.py`
**Lines:** 441-449

**Trigger:** User uploads an Excel file for validation (NEW upload)

**Code:**
```python
if not session_id:
    logger.info("No session_id provided - generating new session ID for new table upload")
    import hashlib
    timestamp = datetime.now().strftime('%Y%m%d_%H%M%S')
    hash_input = f"{email_address}_{timestamp}".encode()
    short_hash = hashlib.md5(hash_input).hexdigest()[:8]
    new_session_id = f"session_{timestamp}_{short_hash}"
    base_session_id = new_session_id
    logger.info(f"Generated new session ID for Excel upload: {base_session_id}")
```

**Request:**
```json
{
  "action": "upload",
  "email": "user@example.com",
  "file": "<base64-encoded-excel>",
  "filename": "companies.xlsx"
}
```

**Response:**
```json
{
  "success": true,
  "sessionId": "session_20251022_143530_b2c4d6e8",
  "excelS3Key": "users/user@example.com/sessions/session_20251022_143530_b2c4d6e8/uploads/companies.xlsx"
}
```

**Critical Comment (frontend line 6373):**
> "New file upload should NEVER reuse existing session - always generate new session ID"

This ensures each table upload gets its own isolated session.

## When Session IDs Are Reused

### Existing Session ID Must Be Provided

**Table Maker Continue:**
```json
{
  "action": "continueTableConversation",
  "session_id": "session_20251022_143022_a7b3c9d1",  // REQUIRED
  "conversation_id": "table_conv_...",
  "user_message": "Add more columns"
}
```

**Validation Request:**
```json
{
  "action": "validate",
  "session_id": "session_20251022_143530_b2c4d6e8",  // REQUIRED
  "email": "user@example.com",
  "config_s3_key": "path/to/config.json"
}
```

**Config Upload to Existing Session:**
```json
{
  "action": "upload_config",
  "session_id": "session_20251022_143530_b2c4d6e8",  // REQUIRED
  "email": "user@example.com",
  "config_file": "<base64>"
}
```

## Backend Generation Logic

### Common Pattern

All session ID generation follows this pattern:

```python
import uuid
from datetime import datetime

timestamp = datetime.now().strftime('%Y%m%d_%H%M%S')
random_hex = uuid.uuid4().hex[:8]  # 8 characters from UUID
session_id = f"session_{timestamp}_{random_hex}"
```

### Variations

**Excel Upload (uses MD5 hash):**
```python
import hashlib

timestamp = datetime.now().strftime('%Y%m%d_%H%M%S')
hash_input = f"{email_address}_{timestamp}".encode()
short_hash = hashlib.md5(hash_input).hexdigest()[:8]
session_id = f"session_{timestamp}_{short_hash}"
```

**Why MD5?** Deterministic for email + timestamp combo (easier debugging)

**Demo Sessions (UUID random):**
```python
unique_id = str(uuid.uuid4())[:8]
session_id = f"session_demo_{timestamp}_{unique_id}"
```

## Frontend Behavior

### Never Generates Session IDs

The frontend **NEVER** generates session IDs. It only:

1. **Receives** session_id from backend responses
2. **Stores** in memory and localStorage
3. **Sends** in subsequent requests

### Storage Pattern

```javascript
// After backend response with session_id
currentSessionId = data.session_id;  // Global variable
localStorage.setItem('sessionId', data.session_id);  // Persistent

// On page load
currentSessionId = localStorage.getItem('sessionId');  // Restore

// In subsequent requests
fetch(API_ENDPOINT, {
    body: JSON.stringify({
        action: 'validate',
        session_id: currentSessionId,  // Reuse existing
        email: userEmail,
        ...
    })
});
```

### New vs Existing Session Logic

**New Session (omit session_id):**
```javascript
// Table Maker - new conversation
fetch(API_ENDPOINT, {
    body: JSON.stringify({
        action: 'startTableConversation',
        email: userEmail,
        user_message: userMessage
        // NO session_id field
    })
});

// Excel Upload - new file
fetch(API_ENDPOINT, {
    body: JSON.stringify({
        action: 'upload',
        email: userEmail,
        file: base64Excel
        // NO session_id field
    })
});
```

**Existing Session (include session_id):**
```javascript
// Continue conversation
fetch(API_ENDPOINT, {
    body: JSON.stringify({
        action: 'continueTableConversation',
        session_id: currentSessionId,  // INCLUDE
        conversation_id: currentConversationId,
        user_message: userMessage
    })
});

// Validate with existing config
fetch(API_ENDPOINT, {
    body: JSON.stringify({
        action: 'validate',
        session_id: currentSessionId,  // INCLUDE
        email: userEmail,
        config_s3_key: configKey
    })
});
```

## Session Lifecycle

### Creation
```
Frontend Request (no session_id)
  ↓
Backend: Check if session_id in request
  ↓
if not session_id:
    Generate new session_id
  ↓
Create S3 directory: users/{email}/sessions/{session_id}/
  ↓
Create DynamoDB entry (if applicable)
  ↓
Return session_id to frontend
```

### Usage
```
Frontend stores session_id
  ↓
All subsequent requests include session_id
  ↓
Backend uses session_id to:
  - Locate files in S3
  - Track runs in DynamoDB
  - Associate WebSocket connections
  - Generate download URLs
```

### Persistence
```
S3 Structure:
users/{email}/sessions/{session_id}/
  ├── uploads/          (Excel files)
  ├── configs/          (Validation configs)
  ├── results/          (Validation results)
  ├── downloads/        (Generated files)
  └── session_info.json (Metadata)

DynamoDB:
ValidationRuns table:
  - PK: {email}
  - SK: run#{session_id}#{run_key}
```

## Special Cases

### Preview Mode Sessions

When user previews a table before full validation, a **child session** is created:

**Format:** `{parent_session_id}_preview_HHMMSS`

**Example:** `session_20251022_143530_b2c4d6e8_preview_144015`

**Purpose:** Isolate preview validation from full validation

### Table Maker Execution Sessions

Table Maker uses the **same session_id** throughout:
- Interview conversation
- Table execution
- CSV template generation
- Preview
- Full validation

**Flow:**
```
Session: session_20251022_143022_a7b3c9d1
  ↓
Interview: Multiple turns (same session_id)
  ↓
Execution: Generate table (same session_id)
  ↓
Preview: Show template (same session_id)
  ↓
Validation: Validate data (same session_id)
```

## Debugging Session IDs

### Check Current Session

**Frontend Console:**
```javascript
console.log('Current session:', currentSessionId);
console.log('Stored session:', localStorage.getItem('sessionId'));
```

**Backend Logs:**
```bash
# CloudWatch
aws logs tail /aws/lambda/perplexity-validator-interface --follow --filter-pattern "session_20251022"

# Search for session
grep "session_20251022_143022" /var/log/lambda.log
```

### Common Issues

**Problem:** "No session ID available"
**Cause:** Frontend lost session_id (localStorage cleared)
**Solution:** Upload new file or start new Table Maker conversation

**Problem:** "Session not found in S3"
**Cause:** Session ID exists but no files uploaded yet
**Solution:** Upload Excel file first

**Problem:** Wrong session being used
**Cause:** Frontend reusing old session_id
**Solution:** Clear localStorage or force new session with `session_id: null`

## Best Practices

### Backend

✅ **Always check** `if not session_id:` before generating
✅ **Log generation** with `logger.info(f"Generated new session ID: {session_id}")`
✅ **Return session_id** in response for frontend tracking
✅ **Use consistent format** across all workflows

### Frontend

✅ **Never generate** session IDs client-side
✅ **Store immediately** after receiving from backend
✅ **Omit session_id** for new workflows (let backend generate)
✅ **Include session_id** for continuing existing workflows
✅ **Clear on logout** to force new session
