**Version:** 4.2
**Status:** Production Ready
**Date:** 2026-03-04

## System Overview

The Clone is an intelligent search and synthesis system that:
1. **Routes smartly** - Strategy-based (breadth/depth), answers directly or searches
2. **Batches efficiently** - 5-15 sources per extraction call for shallow strategies
3. **Handles semantically** - Structured verbal handles for citations
4. **Matches perfectly** - Multi-strategy citation resolution with 2-letter anchor fallback
5. **Self-corrects** - Grade-signal loop triggers re-search and re-synthesis when needed
6. **Tracks costs precisely** - Provider-level cost aggregation

## Latest Updates (v4.2)

### Citation Format: 3-Element Bracket
- All citations now use `[handle, snippet_id, AA]` format
- The 2-letter anchor (AA, AB, …) is the **last comma-element inside the bracket**
- LLM copies the full bracket verbatim from snippet listings — no reconstruction needed
- Listing format: `[handle, S1.1.0.0-p0.95, AA] (p=0.95, c=H/P) "text"`

### Two-Letter Anchor System
- Each snippet gets a stable 2-letter short_id (AA, AB, …, ZZ) assigned at synthesis time
- Generated by `_make_short_id(index)` in `unified_synthesizer.py`
- Used as a fast, unambiguous citation anchor (Strategy 0 in resolution pipeline)
- Supports 676 unique IDs per synthesis call (wraps with suffix: AA2, AB2, … beyond 676)

### Citation LLM Resolution
- After strategies 0–4 fail, a batch LLM call resolves remaining unmatched references
- Candidate list now includes `anchor=AA` field so the LLM can match by anchor
- Prompt explains 3-part format and priority: anchor → snippet_id → handle+context
- Model: `gemini-3-flash-preview-low`; logged as independent step in debug summary

### Self-Correction Loop with Grade Signals
- Synthesis self-assesses using `GRADE[|T][|S][|E]` format
- Loop triggers re-search + re-synthesis when grade < A and actionable signals present
- Signals: `|T` (more thinking), `|S` (more search, provide terms), `|E` (escalate model tier)
- Max 1 self-correction by default (more for findall mode)

### Debug Summary Ordering Fix
- `CloneLogger.reserve_step_index()` captures slot before `evaluate_and_synthesize`
- Synthesis/Re-Synthesis metrics inserted at reserved slot so nested steps (e.g. Citation LLM Resolution) appear **after** the synthesis row in the summary table

### `skip_citation_resolution` Parameter
- `evaluate_and_synthesize(..., skip_citation_resolution=True)` defers citation resolution to caller
- Useful for intermediate passes that will be discarded by self-correction (saves cost)
- Infrastructure is in place; call sites in `the_clone.py` can opt in

## Snippet ID Format

```
S{iteration}.{search}.{source}.{quote}-p{score}

Example: S1.2.3.0-p0.85
  = Iteration 1, Search 2, Source 3, Quote 0, p-score 0.85
```

- **Memory snippets:** `SM.{search}.{source}.{quote}-p{score}`
- **Citation recall snippets:** `SC.{search}.{source}.{quote}-p{score}`
- **p-score:** Expected factual accuracy (0.05–0.95)
  - 0.85–0.95: High confidence (PRIMARY/DOCUMENTED/ATTRIBUTED)
  - 0.50–0.65: Medium (OK quality)
  - 0.05–0.30: Low (UNSOURCED/STALE/PROMOTIONAL)

## Two-Letter Anchor System

Each snippet gets a stable 2-letter short_id (AA, AB, AC, …, ZZ) assigned at the start of each `evaluate_and_synthesize` call:

```python
# In unified_synthesizer.py
for i, snippet in enumerate(snippets):
    if 'short_id' not in snippet:
        snippet['short_id'] = _make_short_id(i)  # AA, AB, ..., ZZ
```

The anchor appears as the **third element** in the citation bracket shown to the synthesis LLM:

```
- [mortality_rate_icu, S1.2.3.0-p0.85, AB] (p=0.85, c=H/P) "ICU mortality was 23%..."
```

The LLM is instructed to copy the full bracket verbatim. This makes citation matching robust: even if the handle or snippet ID drifts (e.g., wrong iteration number after self-correction), the anchor alone is enough to identify the snippet.

## Citation Resolution Pipeline

After synthesis, `_convert_snippet_ids_to_citations()` resolves all `[...]` brackets in the answer using a cascade of strategies:

| Strategy | Mechanism | Example |
|----------|-----------|---------|
| 0 | **2-letter anchor** — bare `[AA]` → `short_id_map` | `[AB]` → `S1.2.3.0-p0.85` |
| 1 | **Handle + ID** — parse comma-separated bracket | `[handle, S1.2.3.0-p0.85, AB]` |
| 2 | **Direct snippet ID** — starts with `S` | `[S1.2.3.0-p0.85]` |
| 3 | **Prefix ID match** — shortened/partial ID | `[S1.2.3-p0.85]` → full ID |
| 4 | **Verbal handle** — exact/fuzzy handle lookup | `[mortality_rate_icu_2]` |
| 4b | **Sibling fallback** — same source, any snippet | `S1.2.3` prefix match |
| 5 | **LLM batch resolution** — semantic matching | remaining unmatched refs |

Strategy 0 (anchor) is tried first because it is unambiguous. The LLM batch resolver (Strategy 5) is only invoked for refs that survive all earlier strategies, and its prompt includes the `anchor=XX` field for each candidate so it can match by anchor too.

## Grade Signal System

Synthesis self-assessment uses pipe-separated format: `GRADE[|T][|S][|E]`

```
"A"       → Expert quality, no self-correction needed
"B|S"     → Acceptable but needs more search; provide suggested_search_terms
"C|T|S"   → Insufficient; needs more thinking + more search
"B|E"     → Escalate to a stronger model tier
"A+|T"    → Great answer but thinking signal noted (no action since grade ≥ A)
```

**Signal effects (only active when grade < A):**
- `|T` → bump flash thinking one level (min→low→medium→high); at limit acts as `|E`
- `|S` → re-search using `suggested_search_terms`, then re-synthesize
- `|E` → step up exactly one tier class (tier1/2 → tier3/Sonnet → tier4/Opus)

## Synthesis Input Format

Quotes are grouped BY SEARCH TERM for better synthesis:

```
Q1.1: "Claude Opus 4 architecture" [★ NEW]
  [S1.1.0] https://anthropic.com/claude | Introducing Claude 4
    - [claude4_hybrid_arch, S1.1.0.0-p0.95, AA] (p=0.95, c=H/P) "Uses hybrid architecture with two modes"
    - [claude4_context_200k, S1.1.0.1-p0.95, AB] (p=0.95, c=H/P) "Supports 200K context window"

Q1.2: "GPT-4.5 performance" [★ NEW]
  [S1.2.0] https://openai.com/gpt45 | GPT-4.5 Analysis
    - [gpt45_benchmark_92, S1.2.0.0-p0.85, AC] (p=0.85, c=H/D) "Achieves 92% on benchmarks"
  [S1.1.0] https://anthropic.com/claude | Introducing Claude 4 (off-topic)
    - [claude4_gpt45_params, S1.1.0.2-p0.85, AD] (p=0.85, c=H/P) "[re: GPT-4.5] Comparison shows GPT-4.5 at 12.8T params"
```

**Benefits:**
- Synthesis sees all info about one aspect together
- Easier to organize comparison answers
- Off-topic quotes clearly marked with `[re: ...]` context marker

## Debug Log System (CloneLogger)

Every run produces a structured Markdown debug log:

```
clone_logger = CloneLogger(debug_dir=...)

clone_logger.start_step("Synthesis")          # opens <details> section
...
slot = clone_logger.reserve_step_index()      # capture position in summary table
await synthesizer.evaluate_and_synthesize(...) # may log nested steps (Citation LLM Resolution)
clone_logger.record_step_metric("Synthesis", ..., position=slot)  # inserted at reserved slot
clone_logger.end_step("Synthesis")            # closes section

clone_logger.finalize(metadata, answer, citations)  # inserts summary table at top
```

The `reserve_step_index()` / `position=` pattern ensures that steps logged inside `evaluate_and_synthesize` (e.g., Citation LLM Resolution) appear **after** Synthesis in the summary table, preserving chronological order.

### Parallel Processing

**Triage (Step 2):**
```python
# All search terms triaged concurrently
tasks = [triage(search_1), triage(search_2), ..., triage(search_5)]
results = await asyncio.gather(*tasks)
```

## Optimization Techniques

### 1. Smart Routing (Step 0)
- Answers ~20% of queries directly (no search)
- Saves ~$0.10 and 60-120s per direct answer

### 1b. Skip-to-Synthesis Path
When direct answer fails validation/repair, skip search and go straight to synthesis:
- Extract URLs from query using `extract_urls_from_text()`
- Look up URLs in memory via `SearchMemory.recall_by_urls()`
- Fetch missing URLs live via Jina AI Reader
- Convert URL sources to snippets and synthesize
- Faster than full search when URLs are available in query

**Decision Flow:**
```
answer_directly + valid answer     → Return answer
answer_directly + repairable       → Repair, return answer
answer_directly + unrepairable     → Skip-to-synthesis (with URL sources)
need_search                        → Full search path
```

### 2. Parallel Triage + Extraction
- Concurrent evaluation of all searches/sources
- 5-7x faster than sequential triage
- 3-4x faster per source for extraction (essential quotes only)

### 3. Off-Topic Detection
- Capture cross-search information
- 36% bonus quotes without extra API calls
- Off-topic quotes get `[re: topic]` context markers

### 4. Self-Correction Loop
- Grade signals drive targeted re-search (not blind retry)
- Only re-synthesizes when actionable signals are present
- Escalates model tier only when explicitly signaled (`|E`)

### 5. 2-Letter Anchor for Citation Robustness
- Assigned at synthesis time; unambiguous per-snippet identifier
- Survives handle/ID drift across self-correction iterations
- Resolution Strategy 0: O(1) anchor lookup before any fuzzy matching

## URL Extraction and Memory Recall

### URL Detection
The system automatically detects URLs in user queries using `extract_urls_from_text()`:
- Matches `http://` and `https://` URLs
- Extracts domain and path
- Used for memory lookup and live fetching

### URL-Based Memory Recall
When URLs are found in the query (via `SearchMemory`):

1. **Memory Lookup** (`recall_by_urls()`):
   - Checks if URL sources exist in search memory
   - Returns `{found: [...], not_found: [...]}`

2. **Live Fetch** (`fetch_url_content()`):
   - URLs not in memory are fetched via Jina AI Reader
   - Jina converts web pages to clean markdown
   - Fetched content is converted to source format

3. **Priority Handling**:
   - URL sources get high priority in source selection
   - Always included in synthesis
   - Bypass keyword filtering for direct relevance

## Error Handling

- API failures: Continue with available data
- Empty extractions: Skip sources, continue
- Triage returns 0: Proceed to synthesis with existing snippets
- Schema validation: Soft schemas with fallback parsing
- Citation resolution failures: LLM batch resolver as final fallback; unresolvable refs uploaded to S3 as `snippet_failures`

## Experimental Approaches (Not in Production)

### Merged Synthesis (Code-Based Citations)

**Status**: Experimental - Fully functional but not integrated
**Documentation**: See `MERGED_SYNTHESIS_EXPERIMENTAL.md`

An alternative approach that combines extraction and synthesis into a single LLM call:

```
┌─────────────────────────────────────────────┐
│  All sources → Label with codes (`S1:1.1)   │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  Single LLM Call:                            │
│  - Synthesize from all labeled sources       │
│  - Cite using codes {`S1:1.1, 0.95, P}       │
│  - Include p-scores inline                   │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  Post-processing:                            │
│  - Resolve codes to text                     │
│  - Create citations [1][2]                   │
└─────────────────────────────────────────────┘

Total: 1 LLM call (vs N+1 traditional)
```

**Validated Performance**:
- ✅ 85.7% LLM call reduction (7 → 1)
- ✅ 15-20% token reduction
- ✅ 99-100% code resolution rate
- ⚠️ ~10% higher cost (output token heavy)
- ⚠️ 50-90% slower (single large call)

**Use cases**: Rate-limited scenarios, cost-per-call billing, batch processing

**Current decision**: Stick with traditional approach for production (speed + proven stability)
